# Lesson Authoring Guide

Этот документ описывает, как создавать уроки в рамках текущего курса, чтобы материал был последовательным, практичным и легко усваивался.

## Связь с roadmap разработки

- Каждый урок соответствует **ровно одному шагу** из [`development-roadmap.md`](./development-roadmap.md).
- Номер в имени файла (`lesson-NNN-...`) **должен совпадать** с номером шага NNN.
- Если шаг отменён или разбит — обновляется roadmap (ADR) и создаётся новая нумерация; не оставлять «дыры» без пояснения в roadmap.

## Sprint lesson contract (обязательные секции)

Каждый урок-спринт, кроме секций из шаблона [`lessons/lesson-template.md`](./lessons/lesson-template.md), **обязан** явно содержать блоки (можно как `##` заголовки):

1. **Learning Goal** — что изучаем и зачем.
2. **Implementation Scope** — границы изменений в коде.
3. **Dependencies** — пакеты, сервисы, версии Node и т.д.
4. **Step-by-Step Changes** — воспроизводимые шаги.
5. **Verification** — команды и тесты + ожидаемый результат.
6. **Changed Files** — таблица или список путей.
7. **Architecture Notes** — решения и trade-offs.
8. **Definition of Done** — чеклист приёмки шага.
9. **External operations (outside the repo)** — что делать **вне** редактора кода (см. ниже); обязательно даже если ответ «ничего не нужно».

Секции **Context**, **Concept**, **Code Changes**, **Why This Matters**, **What To Remember**, **Verify**, **Homework** из шаблона **не удалять**; дублирование с _Verification_ допустимо, если _Verify_ даёт краткую выжимку для быстрого прогона.

## External operations (вне репозитория)

Курс рассчитан на разработчика, который **осваивает архитектуру**: не только TypeScript в monorepo, но и Docker, облачные панели, DNS, managed-сервисы. Если шаг требует действия снаружи репозитория — урок **обязан** это проговорить явно, с подсказками «где нажать» и «зачем в системе».

### Обязательная секция в каждом уроке

Заголовок: **`## External operations (outside the repo)`** (можно по-русски: «Вне репозитория» — но единообразно в новых уроках предпочтителен английский заголовок из шаблона).

**Состав секции:**

1. **Таблица «что / где / зачем»** (минимум 1 строка или явная фраза «внешних действий нет»).

   | Действие                 | Где (инструмент)                             | Нужно на этом шаге? | Зачем в архитектуре (1 фраза)                  |
   | ------------------------ | -------------------------------------------- | ------------------- | ---------------------------------------------- |
   | Поднять Postgres         | `docker compose up` в корне, см. LOCAL_SETUP | Да                  | API и миграции живут в контейнере, не на хосте |
   | Создать проект на Vercel | vercel.com dashboard                         | Нет (шаг 3xx)       | SSR `apps/web` задеплоим позже                 |

2. **Architecture sketch** — короткий абзац (3–6 предложений): что выполняется на ноутбуке, что в Docker, что останется для production (Railway / Vercel / Supabase и т.д., если релевантно). Без маркетинга — как для напарника: «браузер → web → api → postgres».

3. **Отложено** — если в шаге только задел (stub, env без реального провайдера), написать **номер будущего шага** из roadmap и что студенту **не нужно** регистрироваться прямо сейчас.

### Категории, которые нельзя «промолчать»

- **Docker / Compose** — какой сервис, порт, URL UI (MailDev, Adminer, Jaeger).
- **Локальные бинарники** — Node версия, `npm ci`, если неочевидно из Track 0.
- **Облако приложений** — Railway, Render, Fly.io: сервис, переменные, связь с `apps/api` / `apps/web`.
- **Хостинг фронта** — Vercel, Netlify: root directory, build command, env `VITE_*` vs server secrets.
- **Managed БД / BaaS** — Supabase, Neon: connection string, отличие от `docker compose` Postgres в dev.
- **Почта / OAuth SaaS** — Resend, SendGrid, Clerk: ключи, redirect URLs (когда появятся в roadmap).
- **CI / секреты** — GitHub Actions: какие secrets добавить вручную в Settings.
- **DNS / TLS / CDN** — домен, CNAME, сертификаты (Track 7–8).

### Если внешних действий нет

Явная формулировка, например:

> **Вне репозитория на этом шаге ничего не требуется:** достаточно monorepo и уже описанного local compose (Postgres). Аккаунты Railway, Vercel, Supabase не создаём.

Не оставлять секцию пустой без такой фразы.

### Связь с другими документами

- Повторяющиеся команды и порты — в [`LOCAL_SETUP.md`](./LOCAL_SETUP.md); в уроке — **ссылка** + только то, что новое для шага.
- **Storytelling** — метафора и «зачем», без кликов по dashboard (см. [storytelling.mdc](../.cursor/rules/storytelling.mdc)).
- **Architecture Notes** — trade-offs кода; **External operations** — где физически крутится система.

Правило для агентов: [`.cursor/rules/external-operations-lessons.mdc`](../.cursor/rules/external-operations-lessons.mdc).

## Core Principles

1. Один урок = одно концептуальное ядро (один шаг roadmap).
2. Минимально достаточная теория, максимум практики.
3. Каждое изменение кода должно иметь учебную цель.
4. Любая новая абстракция объясняется "зачем".
5. Урок должен быть самодостаточным для повторения.

## Narrative Rules

- Писать на русском, технические термины и код - на английском.
- Использовать обязательную структуру: Context -> Concept -> Step-by-Step -> Code Example -> Verification.
- Избегать абстрактных формулировок без примера из текущего кода.
- Для каждого нового понятия давать минимум один рабочий пример кода (TypeScript, DTO, guard, test, или command snippet).
- Добавлять объяснение "как думать" (decision rationale), а не только "что сделать".
- Добавлять "What To Remember" (3-5 тезисов, не больше).

## Lesson Depth Standard

- Урок должен быть достаточным, чтобы студент смог повторить шаг без внешних источников.
- Каждый шаг описывается как:
  - цель шага;
  - конкретное изменение;
  - ожидаемый результат.
- Если в уроке есть архитектурный выбор, нужно указать trade-offs.
- Примеры кода должны быть короткими, но исполнимыми по смыслу.
- В уроках roadmap-уровня обязательно добавлять структуру внедрения по шагам.

## Code Change Rules

- Изменения должны быть малыми и атомарными.
- Нельзя смешивать несколько новых концепций в одном шаге.
- Перед введением новой техники проверять, что базовый сценарий работает.
- При необходимости использовать TDD/test-first ритм: сначала добавляем/обновляем тест (ожидаемо падающий/красный), затем минимальный код чтобы тест стал зелёным, и только после этого рефакторинг без изменения поведения.
- Каждый урок должен содержать команды проверки (`npm run start:dev`, `npm run test`, HTTP checks).
- Если добавляется инфраструктура (DB, cache, queue), в уроке указывать локальный запуск **и** секцию **External operations** (Docker UI, порты, что не нужно в облаке на этом шаге).

## Lesson Quality Checklist

- Понятно, зачем нужен урок?
- Ровно одно ключевое понятие?
- Есть связь с предыдущими уроками?
- Есть явный список изменений в коде?
- Есть хотя бы один код-пример, объясняющий концепцию?
- Есть пошаговый план внедрения (если это roadmap/проектирование)?
- Есть шаги валидации результата?
- Есть секция "что запомнить"?
- В блоках `Verification` и/или `Step-by-Step Changes` явно описан тест-first (TDD) порядок: сначала тест, затем минимальный код, затем рефакторинг (если применяется).
- Для шагов с кодом в `apps/api/src`: есть `*.spec.ts` в том же change set (иначе pre-commit упадёт).
- Есть **Definition of Done** и воспроизводимые команды проверки?
- Есть секция **External operations** (таблица + architecture sketch или явное «внешних действий нет»)?
- Обновлён блок шага в [`storytelling.md`](./storytelling.md)?

## Folder and Naming Convention

- Уроки лежат в `docs/lessons`.
- Формат имени: `lesson-NNN-topic.md`, где **NNN** — трёхзначный номер шага (например `001`, `057`).
- Нумерация строго по [`development-roadmap.md`](./development-roadmap.md).
- Новые уроки создаются **только для шагов из roadmap**; отклонения оформляются ADR и правкой индекса шагов.

## Documentation sync checklist

При **любом** завершённом шаге roadmap синхронизируйте все релевантные поверхности в **том же PR/коммите**, что и код. Не отмечайте шаг `done`, пока индексы и narrative не актуальны.

| Документ                                             | Когда обновлять                                                                                   |
| ---------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| [`docs/lessons/lesson-NNN-*.md`](./lessons/)         | Всегда: новый или обновлённый урок по шаблону                                                     |
| [`development-roadmap.md`](./development-roadmap.md) | **Baseline Status** (срез в начале) + строка шага (`done`) в Track table + таблица уроков в конце |
| [`README.md`](./README.md)                           | Секция Completed Lessons                                                                          |
| [`learning-path.md`](./learning-path.md)             | Ссылка на урок в фазе                                                                             |
| [`storytelling.md`](./storytelling.md)               | **Обязательно** для шагов Track 1+ (см. ниже)                                                     |
| [`LOCAL_SETUP.md`](./LOCAL_SETUP.md)                 | Сменился «следующий шаг» или команды локального запуска                                           |
| Связанные уроки                                      | Обратные ссылки, если шаг закрывает отложенный scope                                              |
| [`docs/README.md`](./README.md) runbook / API        | Только если шаг меняет операторские инструкции                                                    |
| `apps/api/src/**/*.spec.ts`                          | **Обязательно** в том же коммите, что и production-код API — см. ниже                             |

Правила для агентов Cursor:

- [`.cursor/rules/documentation-sync.mdc`](../.cursor/rules/documentation-sync.mdc)
- [`.cursor/rules/storytelling.mdc`](../.cursor/rules/storytelling.mdc) — как писать `docs/storytelling.md` (история, не changelog)
- [`.cursor/rules/tests-first-api.mdc`](../.cursor/rules/tests-first-api.mdc) — pre-commit/CI gate; e2e **не** заменяет unit-тесты
- [`.cursor/rules/external-operations-lessons.mdc`](../.cursor/rules/external-operations-lessons.mdc) — Docker, облако, dashboard: явно в уроке

## Tests-first gate (API)

При любом изменении `apps/api/src/**` (кроме `*.spec.ts`) в **том же коммите** должны быть unit-тесты `apps/api/**/*.spec.ts`. Проверка: [`scripts/validate-tests-first.mjs`](../scripts/validate-tests-first.mjs) (husky pre-commit, CI).

- Колокация: `feature.service.ts` → `feature.service.spec.ts`.
- Обычно тестируем сервис/чистые функции; e2e — дополнение, не замена для gate.
- В уроке: секция **TDD Sequence**, строки в **Changed Files** и **Definition of Done** для каждого `*.spec.ts`.

## Storytelling sync

**Предустановка проекта (обязательна для агентов и авторов):** [`.cursor/rules/storytelling.mdc`](../.cursor/rules/storytelling.mdc) — как писать главы, что запрещено, пример плохо/хорошо. Не сворачивать Track 2+ обратно в одну главу-changelog.

Параллельно с уроком обновляйте [`storytelling.md`](./storytelling.md) — человеческий сюжет «зачем в большой истории», без команд, кода и списков файлов из урока.

**Storytelling не опционален:** пропуск narrative-обновлений считается незавершённым шагом, даже если урок и roadmap уже обновлены.

После каждого спринта:

1. Определите **главу** (см. оглавление в storytelling: I–XIII для текущих треков; для нового сюжета — новая глава, не append в конец одной).
2. Добавьте **одну строку** в таблицу **«Уроки этой главы»**: номер шага, суть одной фразой, ссылка на `lesson-NNN-*.md`.
3. Допишите **2–4 предложения к арке** в блоке **«Как это сложилось у нас»** (или «Как это выглядело» в главах I–IX) — связный рассказ, не цепочка `057 — … 058 — …`.
4. При необходимости уточните **«Связь с прошлым»** (Track 2+) и обновите **«Где мы сейчас»** / **«Что дальше»** коротким checkpoint (без списка эндпоинтов и имён сервисов).
5. Если шаг меняет понимание прошлого — уточните narrative в той главе, не дублируя таблицу и не копируя Verification из урока.

Каркас главы (Track 2+): **Сцена → Что болит → Намерение → Как сложилось → Связь с прошлым → Что унести → Уроки**. Не возвращайтесь к формату «53 отдельных `### Шаг`» и не дописывайте в prose блоки «После NNN …».

### Anti-pattern (плохо / хорошо)

**Плохо** — changelog в narrative одной главы:

> **057** — bootstrap TypeORM … **058** — DATABASE_URL … **063** — POST /auth/register … **083** — CmsModule …

**Хорошо** — арка и связи:

> Сначала заложили память в Postgres и миграции, без автосхемы при старте. Когда появился вход, те же Problem Details, что в Track 1, стали отвечать за 409 при дубликате email. Первый CMS-маршрут — проверка права читать, а не список постов.

Storytelling не заменяет Learning Goal в уроке: урок — «как сделать», storytelling — «зачем и в какой главе истории».

## Template Contract

Каждый новый урок обязан основываться на `docs/lessons/lesson-template.md` и включать все обязательные секции шаблона **без удаления**; дополнительные секции спринта добавляются поверх.
